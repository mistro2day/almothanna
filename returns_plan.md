# خطة تنفيذ نظام المرتجعات (Returns Module Design & Plan)

يوضح هذا المستند التصميم الهيكلي والتفصيلي لإضافة نظام المرتجعات في تطبيق "المثنى للأدوية" بناءً على طلبكم الكريم.

---

## ⚠️ تنبيهات هامة وملاحظات مراجعة
> [!IMPORTANT]
> **اختيار نمط الإرجاع:** يتيح النظام للمستخدم الاختيار التفاعلي بين إرجاع صنف معين بكمية محددة أو إرجاع كامل الفاتورة دفعة واحدة.

> [!WARNING]
> **التأثير المالي وصندوق النقد:** يمكن للمستخدم اختيار إما إرجاع المبلغ نقداً للصندوق (وبالتالي يسجل حركة مالية سالبة في الصندوق اليومي)، أو تخفيض مديونية العميل المتبقية فقط دون إرجاع كاش (خاصة في حالات البيع الآجل).

---

## 🛠️ أولاً: التعديل المقترح على قاعدة البيانات (`schema.prisma`)

سنقوم بإضافة نموذجين جديدين لقاعدة البيانات لتوثيق المرتجعات:

```prisma
model Return {
  id           String       @id @default(uuid())
  saleId       String
  sale         Sale         @relation(fields: [saleId], references: [id], onDelete: Restrict)
  totalRefund  Float        // إجمالي المبلغ المسترجع للعميل
  refundToCash Boolean      @default(true) // هل تم سحب المبلغ وإرجاعه من الصندوق؟
  createdAt    DateTime     @default(now())
  items        ReturnItem[]
}

model ReturnItem {
  id         String   @id @default(uuid())
  returnId   String
  return     Return   @relation(fields: [returnId], references: [id], onDelete: Cascade)
  productId  String
  product    Product  @relation(fields: [productId], references: [id], onDelete: Restrict)
  batchId    String
  batch      Batch    @relation(fields: [batchId], references: [id], onDelete: Restrict)
  qty        Int      // الكمية المرتجعة
  refundRate Float    // السعر الفعلي الذي تم الإرجاع به (سعر البيع الأصلي)
}
```

كما سنقوم بإضافة علاقة المرتجعات في نموذج الفاتورة `Sale` كالتالي:
```prisma
model Sale {
  // ... الحقول الحالية ...
  returns        Return[]
}
```

## ⚙️ ثانياً: منطق العمل بالخلفية (Backend Logic)

سنقوم بإنشاء وحدة برمجة جديدة للمرتجعات `ReturnsModule` تحتوي على النقاط التالية:

### 1. معالجة طلب المرتجع (`POST /returns`)
يستقبل الطلب بالصيغة التالية:
```json
{
  "saleId": "IN-00001",
  "refundToCash": true,
  "items": [
    {
      "productId": "prod-uuid",
      "batchId": "batch-uuid",
      "qty": 5
    }
  ]
}
```

**منطق التنفيذ داخل المعاملة (Transaction):**
1. **التحقق من صحة البيانات:**
   - التأكد من وجود الفاتورة الأصلية `Sale` وأن حالتها ليست `DRAFT`.
   - التأكد من أن الكميات المراد إرجاعها لكل صنف لا تتجاوز الكمية المباعة أصلاً في الفاتورة مطروحاً منها أي كميات تم إرجاعها مسبقاً لهذا الصنف.
2. **تحديث كميات المخزن:**
   - زيادة كمية المنتج في التشغيلة المحددة `Batch` بالكمية المرتجعة.
   - إنشاء حركة مخزنية جديدة `StockMovement` من النوع `IN` بالسبب `"Customer Return"` وتوثيق رقم الفاتورة والمرتجع.
3. **معالجة المبالغ المالية والصندوق:**
   - حساب إجمالي قيمة المرتجع = `مجموع (الكمية المرتجعة * سعر البيع الأصلي في الفاتورة)`.
   - **الخيار الأول (إرجاع المبلغ للصندوق = true):** 
     - يتم تسجيل حركة مالية سالبة (مصروفات إرجاع أو بند سحب) أو تقليل رصيد المقبوضات الفعلي للصندوق.
   - **الخيار الثاني (إرجاع المبلغ للصندوق = false):**
     - يتم إنقاص إجمالي مديونية الفاتورة (أو تعديل حقل `paid` و `total` أو الأقساط المتبقية للعميل) ليعكس القيمة المخفضة دون سحب كاش.
4. **تحديث حالة الفاتورة الأصلية:**
   - إذا تم إرجاع كامل الأصناف، تتغير حالة الفاتورة تلقائياً إلى حالة مرتجع كامل (أو إلغاء).
5. **حفظ سجل المرتجع:**
   - إنشاء سجل في جدول `Return` وجداول `ReturnItem` وتوثيق العملية في سجل الأنشطة `Activity`.

## 🎨 ثالثاً: واجهة المستخدم (UI/UX)

1. **زر الإرجاع في تفاصيل الفاتورة:**
   - إضافة زر "إرجاع أصناف / مرتجع" بجانب أزرار الطباعة والتحميل للفواتير غير المسودة.
2. **نافذة منبثقة تفاعلية (Return Dialog Modal):**
   - تعرض قائمة الأصناف الحالية في الفاتورة مع كمياتها الأصلية.
   - تتيح إدخال الكمية المراد إرجاعها لكل صنف (مع حد أقصى يساوي المباع).
   - توفير زر "إرجاع كامل الفاتورة" لتعبئة كافة الكميات تلقائياً.
   - مفتاح تبديل (Toggle Switch) بعنوان: "إرجاع المبلغ نقداً للعميل (خصم من الصندوق)" أو "تخفيض المديونية والأقساط".
3. **عرض سجل المرتجعات:**
   - إظهار حقل المرتجعات السابقة وتفاصيلها داخل الفاتورة لسهولة المراجعة والتدقيق المالي.

---

## 🧪 رابعاً: خطة التحقق والاختبار (Verification Plan)

### الاختبارات اليدوية:
1. **اختبار الإرجاع الجزئي:** إرجاع صنف واحد من فاتورة تحتوي على 3 أصناف والتحقق من زيادة رصيد المخزن بمقدار الكمية المرجعة فقط، ومراجعة كشف حساب الصندوق.
2. **اختبار الإرجاع الكلي:** اختيار إرجاع كامل الفاتورة والتحقق من عودة كافة الكميات للمخزن.
3. **اختبار خيارات الدفع:** تجربة الإرجاع الآجل (تعديل مديونية العميل) والإرجاع النقدي والتحقق من أرصدة الصندوق بدقة.
